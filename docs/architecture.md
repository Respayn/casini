# Архитектура проекта Casini

## Общая структура проекта

Проект Casini следует принципам Clean Architecture, разделяя код на несколько слоев с четкими границами ответственности. Основные слои архитектуры:

```
casini/
├── app/                    # Laravel приложение (инфраструктурный слой)
├── src/                    # Доменная логика и прикладные сервисы
│   ├── Domain/             # Доменные сущности и интерфейсы
│   ├── Application/        # Прикладные сервисы и use-case'ы
│   ├── Infrastructure/     # Реализации инфраструктурных интерфейсов
├── resources/              # Шаблоны, стили, скрипты
├── routes/                 # Маршруты приложения
├── tests/                  # Тесты
└── docs/                   # Документация
```

## Принципы Clean Architecture

### 1. Доменный слой (Domain Layer)

Содержит бизнес-логику и правила предметной области. Включает:

- **Сущности (Entities)**: Основные бизнес-объекты, такие как `Report`, `Client`, `Project`, `Template`
- **Интерфейсы репозиториев**: Контракты для доступа к данным (например, `ReportRepositoryInterface`)
- **Сервисы домена**: Сложная бизнес-логика, не относящаяся к конкретной сущности
- **Value Objects**: Неизменяемые объекты, описывающие характеристики сущностей
- **Перечисления (Enums)**: Типизированные значения, такие как `ReportFormat`, `ProjectType`

### 2. Прикладной слой (Application Layer)

Содержит бизнес-логику приложения, организованную в виде use-case'ов по паттерну CQRS. Включает:

- **Команды и обработчики команд**: Реализация операций изменения данных (например, `GenerateReportCommand` и `GenerateReportCommandHandler`)
- **Запросы и обработчики запросов**: Реализация операций чтения данных (например, `DownloadReportFileQuery` и `DownloadReportFileQueryHandler`)
- **DTO (Data Transfer Objects)**: Объекты для передачи данных между слоями
- **Интерфейсы сервисов**: Контракты для инфраструктурных сервисов (например, `ReportGeneratorInterface`)

### 3. Инфраструктурный слой (Infrastructure Layer)

Содержит реализации интерфейсов доменного слоя и интеграции с внешними системами. Включает:

- **Репозитории**: Реализации интерфейсов репозиториев доменного слоя (например, `ReportRepository`)
- **Сервисы**: Реализации интерфейсов сервисов прикладного слоя (например, `ReportGenerator`)
- **Модели Laravel**: Реализации репозиториев с использованием Eloquent ORM

## Гайдлайны по расположению логики и файлов

### 1. Добавление новой функциональности

1. **Определите доменную сущность**:
   - Если новая функциональность относится к существующей сущности, добавьте необходимые методы в соответствующий класс в `src/Domain/`
   - Если создается новая сущность, создайте новый класс в соответствующем подкаталоге `src/Domain/`

2. **Создайте интерфейсы репозиториев**:
   - Добавьте интерфейс репозитория в `src/Domain/{Entity}/` если требуется доступ к данным

3. **Реализуйте прикладную логику**:
   - Создайте команды/запросы в `src/Application/{Feature}/`
   - Добавьте обработчики команд/запросов в соответствующие подкаталоги
   - Создайте DTO для передачи данных между слоями

4. **Реализуйте инфраструктуру**:
   - Добавьте реализации репозиториев в `src/Infrastructure/Persistence/`
   - Добавьте реализации сервисов в `src/Infrastructure/`

### 2. Работа с моделями Laravel

- **Модели в `app/Models/`** используются только как реализации репозиториев
- **Не добавляйте бизнес-логику** в модели Laravel
- **Используйте модели только для**:
  - Определения связей
  - Атрибутов
  - scope запросов
  - Реализации интерфейсов репозиториев

### 3. Работа с внешними API

Для интеграции с внешними API используйте следующие подходы:

1. **Создайте интерфейс клиента** в `src/Domain/{Service}/` или в `src/Application/{Service}/Contracts/`
2. **Реализуйте клиент** в `src/Infrastructure/{Service}/` или в `src/Infrastructure/Clients/{Service}/`
3. **Создайте фабрику клиента** в `src/Application/{Service}/Factories/` если требуется
4. **Зарегистрируйте реализацию в сервис-провайдере** в `app/Providers/`

### 4. Работа с данными

- **Используйте DTO** для передачи данных между слоями
- **Не передавайте массивы или stdClass** между слоями
- **Создавайте отдельные DTO** для каждой конкретной операции

### 5. Работа с представлениями

- **Blade-компоненты** находятся в `resources/views/components/`
- **Страницы** находятся в `resources/views/pages/`
- **Layouts** находятся в `resources/views/layouts/`
- **Livewire-компоненты** находятся в `resources/views/livewire/`

## Пример структуры для новой функциональности

Допустим, нужно добавить функциональность управления задачами (Tasks):

```
src/
├── Domain/
│   └── Tasks/
│       ├── Task.php
│       ├── TaskRepositoryInterface.php
│       └── TaskStatus.php
├── Application/
│   └── Tasks/
│       ├── Create/
│       │   ├── CreateTaskCommand.php
│       │   └── CreateTaskCommandHandler.php
│       ├── GetList/
│       │   ├── GetTasksListQuery.php
│       │   └── GetTasksListQueryHandler.php
│       └── Data/
│           └── TaskData.php
└── Infrastructure/
    └── Persistence/
        └── TaskRepository.php
```

## Тестирование

- **Unit-тесты** для доменной логики размещаются в `tests/Unit/Domain/`
- **Feature-тесты** для прикладной логики размещаются в `tests/Feature/Application/`
- **Тесты интеграций** размещаются в `tests/Feature/Integration/`

## Заключение

Следование этим принципам и гайдлайнам поможет поддерживать чистую архитектуру проекта, облегчает тестирование, упрощает внесение изменений и способствует переиспользованию кода.
